# Schedule 定时任务 设计计划

## 背景

- **当前状态**: MailCode 是 "dumb pipe"——只在收到邮件时拉起 AI Agent 回复, 没有任何主动触发能力
- **为什么需要**:
  - 用户需要"每天 9 点查 TODO.md 发邮件给我"、"每小时抓一次 GitHub issues"这类周期性任务
  - 当前只能用系统 cron + 手动拼邮件脚本, 体验割裂且无法复用 Claude 调用栈
  - `docs/design-final/design.md` §34-56 明确把 MailCode 定位为 dumb pipe, 定时任务作为 out-of-band 触发需要新入口

## 设计

### 整体方案

- 新增 `mailcode schedule` 子系统
- 在 `mailcode serve` 守护进程里挂一个 Scheduler 线程
- 按用户配置的 4 类调度 (interval / daily / weekly / monthly) 触发
- 复用现有 `EmailChannel.send_reply` 发结果邮件
- 任务定义存放在独立的 `~/.config/mailcode/schedules.json`, 提供 7 个 CLI 子命令仿照已有 `mailcode session` 风格

### 架构决策

| 维度 | 决策 | 理由 |
|---|---|---|
| 触发动作 | 调 Claude 单轮 + 发邮件给 to_email | 复用 `call_claude(prompt, cwd)`, 不走 IMAP 路由、不写 session |
| 调度类型 | 4 类: `interval_seconds` / `daily` (HH:MM) / `weekly` (day_of_week + HH:MM) / `monthly` (day_of_month 1-31 + HH:MM) | 覆盖 80% 用例且零三方依赖; 不用标准 cron 避免引入解析器 |
| 存储 | 单文件 `~/.config/mailcode/schedules.json`, 模块级 `threading.Lock` + tmp+replace 原子写 | 简单, 锁已足够 (单进程多线程) |
| 进程模型 | 挂到 `mailcode serve` 守护进程作为独立线程, `--once` 模式不启 scheduler | 一套进程, 共享 EmailChannel + 日志 + 配置 |
| 时区 | 本地时间 + ISO8601 带本地 UTC offset, 不暴露 tz 配置 | 与现有 `datetime.fromtimestamp()` 风格一致 |
| 启动错过策略 | skip 不补跑; 用 `run-now` 显式触发补跑 | 避免邮件风暴, 用户有显式控制 |

### 数据流

```
Scheduler 线程 (tick=30s) →
  每轮 reload schedules.json →
  对每个 enabled task 比较 now() vs next_run_at →
  到期 → 调 call_claude(prompt, cwd) →
  EmailChannel.send_reply(to_email, subject, body) →
  写回 last_run_at/last_status/next_run_at
```

### 调度算法

`compute_next_run(spec, after_dt)`:
- `interval`: `after_dt + interval_seconds`
- `daily`: 找 `after_dt` 之后第一个 `time=HH:MM`
- `weekly`: 同 daily 但按 `day_of_week` 跳转 (0~6, 0=今天)
- `monthly`: 试 `(year, month, day_of_month)`, 若该月不存在此日 (2/30、4/31) → 跳过到下个月, 最多找 12 个月

## 风险与注意事项

- **serve 挂了任务失效**: 不做独立 daemon; 收件人每次触发自然感知; v2 候选 systemd timer / launchd plist
- **schedules.json 损坏**: 启动 warn + 空任务列表, 不静默修复; `mailcode schedule validate` 单独诊断
- **多 serve 实例并发**: v1 不支持, 文档说明, 假定 1 台机器 1 个 serve
- **与 session 隔离**: 调度任务不写 session 文件, 不进 `is_session_enabled()` 路由
- **Claude 超时**: 复用 300s, 不暴露 per-task timeout
- **max_concurrent=1 默认**: 任务堆积自动跳过, 与"错过不补跑"语义一致; 运行中任务加 running flag 防重复触发
- **call_claude 抽取迁移风险**: `test_conversation_handler.py` 中所有 `patch.object(ch_module, "call_claude")` 需改为 `patch.object(cr_module, "call_claude")`, 否则会 patch 失败导致回归
